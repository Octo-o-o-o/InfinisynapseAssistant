# Partner SSO 账号整合与计费归属 Playbook

> 特定用法总结：产品接入「使用 InfiniSynapse 登录」并用 Partner API Key 把任务计费记到用户平台账号时，账号模型、key 存储与资源边界的默认做法。端点事实见 `docs/reference/api-index.md` §8 与上游 `upstream-docs/infinisynapse-site/zh/markdown/partner-sso-integration-guide.md`，本文不复述。

## 一句话规则

计费身份 = 平台资源命名空间。用谁的 API Key 发任务，数据源 / RAG / 任务就活在谁的账号里——所以「开发者 key 还是用户 key」必须按**资源集（工作区/项目）在创建时定死**，不能按请求切换；Partner API Key 只在服务端、加密落库、每次登录覆盖刷新。

## 计费模式选型

| 模式 | 谁付积分 | 适用 | 代价 |
| --- | --- | --- | --- |
| 自持 key（默认） | 开发者 | 免登录 demo、试用、评委动线 | 必须配防滥用：全局日限 + per-user 日配额 + 归属校验，否则任何拿到 ID 的人都能烧你的积分 |
| Partner SSO 仅登录 | 开发者 | 想复用平台账号体系但成本自担 | 同上；只解决身份，不解决成本 |
| Partner SSO + `withApiKey:true` | **用户本人** | 重度使用、商业化主路径 | 资源命名空间随 key 切到用户账号（见下节铁律）；新用户有注册赠分，天然试用额度 |

判断入口：自有账号 + 自持 key 的常规产品不需要接 SSO；出现「用户量 × 任务成本失控」或「用户要求任务可在自己平台后台审计」时再上第三种。

## 资源边界铁律（实测踩坑总结）

平台的数据库数据源（`/api/ai_database/*`）、RAG（`/api/ai_rag_sdk/*`）、任务（`/api/ai_task/*`）都是**账号私有**：用户 key 发的任务看不到开发者账号注册的数据源 ID，反之亦然。

- key 的选择粒度 = 一组会互相引用的资源（数据源 + RAG + 任务）。落库为资源集级字段（如 `workspace.key_mode: platform | user`），**创建时定死，之后不切**。
- 已在开发者账号注册过数据源的存量资源集不要「迁移」到用户 key——凭据不在你手里（瞬时过境），正确做法是引导用户在新资源集里重新录入注册。
- 同一次任务链路（注册数据源 → 建档 → 固化 RAG → 发核查 → askResponse 追问 → 产物回读/messagePayload → 删除任务）必须**全程同一把 key**。漏一处（常见：产物回读、setShare、删除）就是跨账号 404。

## Partner API Key 处理规范

- 兑换（`withApiKey:true`）后**服务端加密落库**（如 AES-256-GCM，密钥走环境变量）；不进前端、不进日志（扫描器 `INF-SEC-001/002` 语义同样适用于用户 key）。
- 官方语义「同一用户+同一应用固定同一把」→ 每次 SSO 登录**直接覆盖保存**，不做多 key 管理。
- 签发可能失败（用户 key 数达上限，默认 20 把）：绑定流程要能"绑定成功但无 key"降级——资源集回落开发者 key + 提示用户清理平台 key 列表后重绑。
- 解绑 = 清除本地密文；受影响的用户计费资源集要明示"不能再发新任务，已有结论不受影响，重绑即恢复"，不要静默回落到开发者 key（那等于把成本悄悄转回自己）。

## 本地账号整合（防接管）

- 绑定键用 `data.user.id`（官方建议；邮箱/昵称会变）。
- **SSO 回传邮箱撞上已有密码账号时不要自动合并**——平台侧邮箱验证状态未知，自动合并可能被用于接管。引导「先用密码登录，再到设置页绑定」。
- 平台可能不回传邮箱（手机号注册的用户）：本地 email 唯一约束要能容纳占位值。
- `state` 存服务端 session（不要放 URL/localStorage），回调先校验再兑换；SSO 接口域是 `api.infinisynapse.cn/api`，任务接口域是 `app.infinisynapse.cn`——两套 base URL 分开配置。
- 建议整条 SSO 能力挂 feature flag（`SSO_ENABLED` + 凭证齐备才开），代码可先上线、凭证就绪再开。

## 计费归属如何验证（提交前自查）

用两把 key 分别调 `GET /api/ai_task/list`：用户 key 发的任务应**只**出现在用户账号列表（来源标记 API-Key 任务）、**不**出现在开发者账号列表。这一步用一个最小任务即可（积分节制），比看账单直观且可自动化。

## 检查清单

- [ ] key 选择按资源集创建时定死，无请求级切换路径？
- [ ] 任务全链路（含产物回读、追问、删除、分享）同一把 key？
- [ ] Partner key 加密落库、不进日志、登录覆盖刷新、有 20 把上限降级？
- [ ] SSO 邮箱冲突走「先登录后绑定」而不是自动合并？
- [ ] 解绑后用户计费资源集有明确的不可用提示？
- [ ] 用双 key 任务列表验证过一次计费归属？
